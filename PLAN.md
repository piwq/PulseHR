# PulseHR — План MVP · Хакатон «Код рождения» (08–10.06.2026)

> Кейс: **PulseHR** — цифровой сервис опросов сотрудников.
> Цель: **1 место.** Не «много технологий», а решённый кейс + демо, которое не падает + один чёткий вау-момент.
> Этот документ — основа, по которой пишем код. Доменные детали уточняются по ТЗ (помечено `[уточнить по ТЗ]`), архитектура — фиксирована.

---

## 0. Главная идея (за что берём первое)

**Замкнутая петля, а не «ещё один Google Forms»:**

```
опрос → метрика отдела просела → LLM находит причину из свободных ответов
      → алерт руководителю: TG-пуш (телефон жужжит) + тост на дашборде (SSE)
```

Это и есть вау-момент демо. Всё остальное — обвязка вокруг него.

## 1. Итоговая архитектура

Всё в одном `docker-compose`, демо крутится локально на ноуте капитана.
`gateway` = **nginx как reverse-proxy** (единая точка входа, никакого отдельного gateway-продукта).

```

                                     │   gateway / reverse-proxy    │
                                     └──────────────┬───────────────┘
                                                    │
      ┌──────────────────────────────┬─────────────┴──────────────┬─────────────────┐
      │ /api/                        │ /api/insights/              │ /               │
      ▼                              ▼                             ▼                 │
 ┌──────────┐                  ┌──────────────┐             ┌────────────┐           │
 │   core   │  HTTP (sync)  ─► │   insights   │             │  frontend  │           │
 │ Django+  │ ◄────────────── │  Django+DRF  │             │   Vue 3    │           │

**Сервисы в compose:** `nginx`, `core`, `insights`, `bot`, `db (postgres)`, `frontend`.

### Что НЕ берём (сознательно отрезано)
`Redis` · `MinIO` · `Celery` / брокеры · `WebSockets` · `Django Channels` · отдельный gateway-продукт · сложные роли/права · экспорт PDF/Excel · мультиязычность · реальная интеграция с HRIS.
Каждый из них = точка отказа на сцене с нулевым профитом для оценки.

## 2. КЛЮЧЕВАЯ развилка: «микросервисы», но монорепо

**Решение: один Django-проект (одна кодовая база, одни модели, одна миграция) → запускается как несколько контейнеров за nginx.**

Почему именно так, а не отдельные репы на сервис:
- Две отдельные Django-кодовые базы на **общую** Postgres = ад с миграциями и `managed=False`-моделями. Это сожрёт половину хакатона на плюминг — ровно то, чего избегаем.
- Монорепо даёт **топологию микросервисов** (отдельные контейнеры `core`/`insights`/`bot` за gateway, разделяемые и масштабируемые), но модели и миграции — единые. `migrate` гоняется один раз.
- `core` и `insights` поднимаются из **одного образа**, разница только в маршрутизации nginx (какие URL куда). `bot` — тот же образ, другая команда.

**Что говорим жюри:** «На MVP — монорепо и общая БД. В целевой архитектуре — раздельные репозитории и БД на сервис.» Это честно и защищаемо.

Раздельные БД на сервис, отдельные репы, брокер между сервисами — всё это идёт **только картинкой target-архитектуры в питч-дек**, не в код.

## 3. Модель данных (ядро)

Живёт в `apps/surveys` (источник схемы), `Insight` — в `apps/insights`.

| Модель | Поля |
|---|---|
| `Survey` | id, title, description, created_at, is_active |
| `Question` | id, survey_id (FK), text, qtype (`scale`/`text`), order |
| `Response` | id, survey_id (FK), department, submitted_at — **без user_id (анонимность)** |
| `Answer` | id, response_id (FK), question_id (FK), value_num (nullable), value_text (nullable) |
| `Insight` | id, survey_id (FK), department (nullable), summary, severity (1–3), created_at, **tg_sent (bool, default False)** |

`department` — пока `CharField` `[уточнить по ТЗ: список отделов]`.
Stretch: таблица `Department(name, tg_chat_id)` для маршрутизации алертов по руководителям.

### Метрики (вычисляемые, не хранимые)
- **Индекс вовлечённости** = средний балл по `scale`-вопросам, агрегат по отделу и по времени (по `submitted_at`).
- **eNPS** (stretch): отдельный вопрос 0–10 → %промоутеров(9–10) − %критиков(0–6).
- **Тренд** — точки по волнам опроса. На демо историю **сидим скриптом** (см. §7).
- **Триггер алерта (severity):** правило — средний балл отдела `< порога` ИЛИ падение `≥ X%` к прошлой волне → severity 2; сильнее → 3. Текстовую причину к нему добавляет LLM из свободных ответов.

### Анонимность (фича + аргумент жюри)
Агрегаты по сегменту отдаём **только при N ≥ 5 ответов**. Иначе по отделу из 2 человек можно вычислить конкретного. Гейт зашит в `stats`.

## 4. API (минимум)

**core** (`/api/`)
- `GET  /api/surveys/`, `GET /api/surveys/{id}/`
- `POST /api/surveys/{id}/submit/` — батч ответов → создаёт `Response` + `Answer`-ы
- `GET  /api/surveys/{id}/stats/` — агрегаты: средний балл по отделам, точки тренда, eNPS; **гейт N≥5**
- создание/редактирование опросов — **через Django admin** (не пишем UI руками)

**insights** (`/api/insights/`)
- `POST /api/insights/analyze/` — body `{survey_id, department?}` → тянет текстовые ответы → LLM → JSON `{summary, top_themes, severity}` → пишет `Insight`. **+ fallback** (см. §8)
- `GET  /api/insights/{survey_id}/` — список инсайтов
- `GET  /api/insights/alerts/stream/` — **SSE** (отдаёт `Insight` с severity ≥ 2)

**bot** — без HTTP. Луп каждые ~3с: `Insight.filter(severity>=2, tg_sent=False)` → `send_message` → `tg_sent=True`.
Outbound-only (PTB или requests), обработку входящих апдейтов для MVP не делаем. MVP: один `MANAGER_CHAT_ID` из env (твой телефон → жужжит на сцене). Stretch: chat_id по отделу.

### SSE — два подводных камня (зашить сразу)
- **nginx**: на `/api/insights/alerts/stream/` → `proxy_buffering off;`. На стороне Django ответ отдаёт заголовок `X-Accel-Buffering: no` и `Cache-Control: no-cache`. Иначе прокси копит буфер и стрим не течёт.
- **gunicorn**: каждое открытое SSE-соединение держит воркер (генератор поллит БД и спит). Контейнер `insights` запускаем с 3–4 воркерами (или gthread). Pub/sub не нужен — опрос БД переживает несколько воркеров сам.

## 5. Стек

- **Backend:** Django + DRF, gunicorn. `dj-database-url` + `DATABASE_URL` (pg ↔ sqlite одной переменной).
- **DB:** Postgres (контейнер). **SQLite-fallback наготове** — если на площадке Docker/сеть подведут, переключаем `DATABASE_URL` и не теряем демо.
- **Frontend:** Vue 3 + Pinia + chart.js/recharts. На демо — **сборка в статику**, отдаётся через nginx (не гонять dev-server на сцене).
- **LLM:** YandexGPT **или** локальный Ollama `qwen2.5-coder`. Доступ с площадочной сети проверить заранее (часто блочат) → если YandexGPT не пускают, fallback на Ollama локально.
- **TG:** python-telegram-bot (outbound).
- **Инфра:** docker-compose, nginx, всё на ноуте.

## 7. Поэтапный план (привязка к программе)

- **DoD:** `docker compose up` → nginx отдаёт фронт, `/api/surveys/`=200, `/api/insights/alerts/stream/` стримит, bot коннектится к TG.

### Фаза 1 — Ядро (старт 14:00 → чек-поинт 18:00)

### Фаза 2 — Вау-петля (19:00–20:00 + вне площадки)

**Никаких новых фич.**

## 8. Устойчивость демо (критично)

| Риск | Митигация |
|---|---|
| LLM недоступен на площадке | Зашить **rule-based fallback** в `analyze` (топ-слова из негативных ответов → шаблонное summary), проверить сеть в Фазе 0 |
| nginx буферит SSE | `proxy_buffering off;` + `X-Accel-Buffering: no` |
| gunicorn-воркер занят стримом | 3–4 воркера на `insights` |
| миграции между сервисами | монорепо, одна кодовая база (§2) |
| Docker/сеть лягут | SQLite-fallback через `DATABASE_URL` |
| нет данных для тренда | `seed.py` с backdated timestamps |
| демо падает вживую | 3 прогона + резервный скринкаст |

**`analyze` на демо — по кнопке, не авто.** Так презентующий контролирует момент вау («сейчас запускаю анализ отдела продаж…» → клик → телефон жужжит). Авто-триггер — только stretch.

## 9. Сценарий демо (то, что реально берёт балл)

1. **Проблема:** «Руководитель узнаёт о выгорании отдела, когда люди уже пишут заявления.»
2. Дашборд: отдел продаж, вовлечённость падает 3 волны подряд (seed).
3. «Запускаем пульс-опрос» — пара ответов вживую (или kiosk-режим).
4. «Система анализирует свободные ответы» — клик `analyze`.
5. **Bam:** на дашборде всплывает тост «Отдел продаж: критично — топ-причина: переработки», **телефон жужжит** — «руководителю уже улетел алерт в телеграм с причиной».
6. «Замкнутая петля: опрос → инсайт → действие, без аналитика-человека.»
7. Слайд **target-архитектуры**: «вот так это раскладывается на сервисы при масштабе» (микросервисы + раздельные БД — картинкой).

**Структура дека:** проблема → кто страдает (HR/руководитель) → решение → live-демо → impact/ROI → target-архитектура → команда.

## 11. Резюме развилок («почему так»)

- **Монорепо вместо реп-на-сервис** — избегаем миграционного ада на общей БД; топология микросервисов сохранена.
- **SSE вместо WebSockets** — поток однонаправленный (сервер→дашборд), не тащим Redis/Channels.
- **bot поллит БД вместо HTTP от insights** — нулевая связанность, переживает рестарты.
- **`analyze` по кнопке на демо** — контроль момента вау.
- **SQLite-fallback** — страховка демо.
- **Postgres + Docker** — солидность для жюри; Docker оправдан именно ради воспроизводимой БД у всей команды.
- **MinIO/Redis/Celery/WS — отрезаны** — точки отказа без профита для оценки.
```
